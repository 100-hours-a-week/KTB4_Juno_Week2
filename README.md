# 🏐 (하이큐) 카라스노 고교 선수 육성 시뮬레이션 🏐

[기존의 동기 CLI 프로그램](https://github.com/100-hours-a-week/KTB4_Juno_Week1)에 **스레드 기반 기능을 추가**하여 **프로그램 안에서 별도의 스레드가 동작**하도록 수정하였다.

## 추가된 기능

### 1. 경기 시작 전 카운트다운 스레드 구현

기존에는 경기 상대를 선택하면 바로 전력 계산과 승패 출력이 되었으나, 기능 추가 후 **경기 시작 전 별도의 스레드에서 카운트다운이 실행**되도록 구현하였다.

```java
Thread countdownThread = new CountdownThread();
countdownThread.start();

try {
    countdownThread.join();
} catch (InterruptedException e) {
    System.out.println("경기 시작 대기 중 문제가 발생했습니다 (꒦ິ⍸꒦ິ)");
    Thread.currentThread().interrupt();
    return;
}
````

- 위 코드에서 `start()`를 사용하여 새로운 스레드 생성을 요청하였다.

  - `run()`을 직접 호출할 경우에는 새로운 스레드가 생성되지 않고 일반 메서드 호출처럼 현재 스레드에서 실행된다. 따라서 `start()`를 호출하여 새로운 스레드를 만들고, 그 안에서 `run()`이 실행되도록 구현하였다. 그 결과 카운트다운은 `main` 스레드의 흐름과 분리된 별도의 스레드에서 동작하도록 하였다.

### 2. 응원 스레드 상호작용 기능 구현

- `KarasunoCheerThread`와 `EnemyCheerThread`는 하나의 `CheerBoard` 객체를 공유한다. 카라스노 응원 스레드는 `CheerBoard`에 카라스노 응원 점수를 추가하고, 상대 학교 응원 스레드는 같은 `CheerBoard`에 상대 학교 응원 점수를 추가하도록 구현하였다.

  - `CheerBoard`는 두 스레드가 함께 접근하는 공유 객체이기 때문에 한 스레드가 응원 점수를 변경하는 동안 다른 스레드는 잠시 기다리도록 `synchronized`를 사용하였다. 이를 통해 두 스레드가 동시에 같은 객체의 값을 수정하는 문제를 줄이고자 하였다.
  - 또한 응원 점수를 읽는 `getKarasunoCheerScore()`와 `getEnemyCheerScore()`에도 `synchronized`를 사용하여 공유 데이터를 안전하게 읽을 수 있도록 구현하였다.
- 기존 `EnemySchool`은 학교에 해당하는 번호, 학교 이름, 전력만 관리하고 있었는데 응원 스레드 기능을 추가하면서 각 학교별 응원 문구도 함께 관리하도록 수정하였다.

  - 응원 문구도 상대 학교와 관련된 고정 데이터이므로 enum에서 함께 관리하는 것이 적합하다고 판단하였다. 그래서 `EnemyCheerThread`에서 `enemySchool.getCheerMessage()`를 통해 해당 학교의 응원 문구를 출력하도록 하였다.

## 회고

### 1주차 과제 코드리뷰 반영 리팩토링 회고

- 1주차 과제의 코드리뷰 피드백을 반영하면서 가장 크게 수정한 부분은 **객체 관리 방식**이다.

  - 기존에는 번호를 기준으로 멤버나 선수를 직접 분기하는 방식이 많았으나, 해당 프로그램은 사용자가 입력한 번호와 객체를 연결하는 방식이 더 적합하다고 판단하였다. 그래서 `List`와 `Map` 중 `Map` 방식으로 코드 구조를 리팩토링하였다.
  - 처음에는 상세 조회 기능에서 `instanceof` 연산자를 통해 `Player`인지 `Manager`인지 구분하려고 했다. 하지만 그렇게 구현하면 `Main`이나 기능 메서드가 객체의 구체적인 타입을 계속 확인해야 하는 번거로움이 있다고 판단하였다. 그래서 `KarasunoMember` 클래스에 `showDetail()`이라는 공통 메서드를 만들고, `Player`와 `Manager`가 **각각 오버라이딩하는 방식으로 구현**하였다.

    ```java
    KarasunoMember member = members.get(memberNumber);
    member.showDetail();
    ```

    이렇게 수정하니 `Main`에서는 해당 멤버가 선수인지 매니저인지 직접 구분하지 않고 `member.showDetail();`만 호출할 수 있게 되었다.

- 기존에는 멤버 정보를 코드에서 직접 순서대로 나열하거나 번호에 따라 분기하는 방식이었는데 피드백을 반영하여 사용자가 입력한 번호와 멤버 객체가 연결되는 구조로 만들기 위해서 HashMap을 사용하였다. 그런데 이 프로그램에서는 순서가 보장되는 `LinkedHashMap`을 사용하는 것이 목록 출력이나 상세 조회에서도 순서가 유지될 것 같아서 `LinkedHashMap`을 사용할 수 있었다.

- 조건문에 따른 메뉴 선택 처리도 기존에는 `if-else` 중심이었지만 `Map<Integer, Runnable>`을 사용해서 메뉴 번호와 실제 실행할 동작을 직접 연결할 수 있도록 구현하였다.

  - `Runnable`을 통해 각 메뉴 번호에 실행할 기능을 미리 연결해두면 입력된 번호로 `menuActions.get(number)`를 찾고 `run()`만 호출하면 된다. 이렇게 구현하면 긴 조건문을 줄이고 메뉴 번호와 기능의 연결 관계를 더 명확하게 표현할 수 있어서 이 방식이 더 적합하다고 생각하였다.

- `MemberNumber`, `MenuNumber`처럼 상수만 모아둔 클래스에는 `private` 생성자를 둘 수 있다는 점도 알게 되었다.

- `람다`를 사용해서 메뉴를 선택했을 때 실행할 코드를 바로 전달할 수 있도록 리팩토링도 진행하였다. 람다는 기존에 자바스크립트 프로그래밍을 할 때 사용했던 문법이라 익숙한 느낌이 있었는데 Java에서도 람다 문법을 사용하면 별도의 클래스를 만들지 않고도 실행할 동작을 간단하게 전달할 수 있어 코드가 더 간결해졌다.

- 훈련 수치도 처음에는 `train()` 메서드에서만 사용되는 값이라 `10`, `6`, `8`이라는 숫자 그대로 선언했다. 하지만 해당 숫자들이 단순한 값이 아니라 훈련 규칙이라는 의미를 지닌 숫자이기 때문에 코드 가독성을 위해 상수로 리팩토링하였다.

### 2주차 과제 구현 회고

- 스레드에 관해서 개념적으로는 이해하고 있었지만, 막상 구현할 수 있을지에 대해서는 자신이 없었다. 하지만 강의 자료를 보면서 `Thread` 클래스를 만들고 `start()`, `sleep()`, `join()`을 사용해보니 스레드가 어떤 식으로 정의되고 실행되는지 더 잘 이해할 수 있었다.

  - 특히 `start()`와 `run()`의 차이를 체감할 수 있었다. `run()`을 직접 호출하면 새로운 스레드가 생성되지 않고 일반 메서드처럼 실행되지만 `start()`를 호출하면 새로운 스레드가 생성되고 그 안에서 `run()`이 실행된다. 따라서 스레드 구현을 보여주기 위해서는 `start()`를 사용해야 한다는 점을 알 수 있었다.

- 카운트다운 스레드를 구현할 때 처음에는 `System.out.println("3...")`, `System.out.println("2...")`... 처럼 직접 출력하는 방식도 생각하였다. 하지만 이렇게 작성하면 숫자가 거의 동시에 출력되어 실제 카운트다운처럼 보이지 않는 문제가 있었다. [참고링크](https://cano721.tistory.com/165)

  - `Thread.sleep()`을 사용해서 현재 실행 중인 카운트다운 스레드를 1초씩 멈추도록 하였고 이를 통해 1초 간격으로 카운트다운이 실행되는 것처럼 보이도록 구현하였다.
  - 현재 구현한 카운트다운 스레드에서는 중단 요청이 들어올 일이 크게 없다고 생각하였다. 하지만 `Thread.sleep()`은 문법상 `InterruptedException`을 발생시킬 수 있으므로 예외 처리가 필요하였다. 이때 `Thread.currentThread().interrupt();`를 사용해 현재 스레드가 중단 요청을 받았다는 상태를 다시 표시하도록 하였다.
  - `System.out.println("3 ..."); Thread.sleep(COUNTDOWN_DELAY_TIME);`와 같은 코드가 반복되어서 `count` 변수를 사용하는 반복문으로 코드 중복을 줄이고자 하였다.
  - 카운트다운 시작 숫자와 카운트다운 대기 시간은 처음에는 일반 숫자로 작성하였다. 하지만 코드의 의도를 더 명확하게 드러내는 것이 가독성에 좋다고 판단하여 **상수**로 분리하였다.
  - 또한 `join()` 메서드를 사용해서 카운트다운이 끝나기 전에 경기 결과가 출력되지 않도록 구현하였다.

- 응원 기능을 추가하면서 스레드 간 상호작용을 더 명확히 이해할 수 있었다. 처음에는 카라스노 응원과 상대 학교 응원을 각각 출력하는 정도로만 생각했지만, 단순 출력만으로는 스레드 간 상호작용이라고 하기에는 부족하다고 느꼈다.

  - 두 스레드가 하나의 `CheerBoard` 객체를 공유하도록 만들었다. 두 응원 스레드는 각각 응원 점수를 추가하고, 그 결과가 경기 전력에 보너스로 반영되도록 구현하였다. 이 과정에서 공유 객체에 여러 스레드가 접근할 때는 동시성 문제를 고려해야 한다는 점을 알게 되었고, `synchronized`를 사용하여 점수 변경 메서드에 한 번에 하나의 스레드만 접근하도록 하였다.

- 구현을 하다 보니 응원 스레드의 실행 순서가 작성한 코드 순서대로 고정되지 않고, 카라스노 응원과 상대 학교 응원이 번갈아 출력되는 것을 확인할 수 있었다. 이를 통해 여러 스레드의 실행 순서는 JVM과 OS의 스레드 스케줄러에 의해 달라질 수 있다는 점을 알 수 있었다.

- 구현 과정에서 `CHEER_DELAY_TIME`을 조정하면 응원 문구가 출력되는 타이밍에도 영향을 줄 수 있다는 점을 알 수 있었다. 여러 조정 과정을 통해 확인해보니 이 값이 출력 순서를 보장한다기보다는 **한 스레드가 멈춰있는 동안 다른 스레드가 실행될 기회를 주는 역할**에 가깝다고 이해하였다.

- `join()` 메서드는 단순히 “기다리는 코드”가 아니라 여러 스레드의 **실행 완료 시점을 맞춰 프로그램 흐름을 제어**하는 데 중요하다는 점을 알게 되었다. 처음에는 아래와 같이 코드를 작성하였다.

  ```java
  try {
    karasunoCheerThread.join();
    System.out.println();
    enemyCheerThread.join();
    System.out.println();
  } catch (InterruptedException e) {
    System.out.println("응원 진행 중 문제가 발생했습니다 (꒦ິ⍸꒦ິ)");
    Thread.currentThread().interrupt();
    return;
  }
  ```

  하지만 `karasunoCheerThread.join()`이 끝났다고 해서 `enemyCheerThread`까지 함께 종료된 것은 아니었다. 카라스노 응원 스레드가 먼저 종료되면 상대 학교 응원 스레드는 아직 실행 중일 수 있는데 그 사이에 `System.out.println();`이 먼저 실행 수도 있었다.

  그래서 두 응원 스레드가 모두 종료된 뒤에 줄바꿈을 처리하도록 아래와 같이 수정하였다.

  ```java
  try {
    karasunoCheerThread.join();
    enemyCheerThread.join();
  } catch (InterruptedException e) {
    System.out.println("응원 진행 중 문제가 발생했습니다 (꒦ິ⍸꒦ິ)");
    Thread.currentThread().interrupt();
    return;
  }

  System.out.println();
  ```

  이를 통해 두 응원 스레드의 실행이 모두 끝난 뒤 다음 출력이 이어지도록 흐름을 더 명확하게 할 수 있었다.
